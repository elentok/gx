# ADR 0012 — Marketing website lives in-repo, deployed standalone to Cloudflare Pages

## Status
Accepted

## Context

`gx` is being published, which needs a public landing page at `gx.elentok.com`. The page is a static
one-pager whose centerpiece is animated demos of the TUI. Two facts shape where it should live:

1. **The demos are intrinsically tied to the binary.** The page is carried by VHS (`.tape`) demo
   recordings, which can only be rendered by running the actual `gx` binary against a seeded git
   repo. The tapes and their generated GIFs are build artifacts of *this* repo, not a separate site.
2. **The README and the website want the same assets.** Both embed the same demo GIFs (single source
   of truth, regenerated by one `make demos`). If the site lived elsewhere, the GIFs would have to be
   copied or submoduled across repos and would drift.

The reference precedent (`~/dev/cryptowl`) is a *standalone* repo deployed with a manual
`wrangler pages deploy`. Following that precedent literally would mean a second repo.

## Decision

The website lives **in the `gx` repo** under `web/`, but is **deployed standalone** to Cloudflare
Pages (not Git-connected).

- `web/` is a self-contained Vite + vanilla-TypeScript project with its own `package.json`. It does
  not participate in the Go module or `make` build except via a `make demos` target that regenerates
  the shared GIFs.
- Deploy is **manual**, mirroring cryptowl: `cd web && npm run deploy`
  (`wrangler pages deploy dist --project-name=gx`). No CI/Git-connected build.
- The custom domain `gx.elentok.com` is configured once in the Cloudflare Pages dashboard.

## Considered Options

- **Separate repo (cryptowl parity).** Rejected: the demo GIFs are artifacts of the `gx` binary, so a
  separate repo would have to import them across a boundary and keep them in sync. The whole point of
  the demos is "regenerate the tape → rebuild the site" as one workflow.
- **In-repo + Git-connected auto-deploy.** Rejected for now: push-to-deploy would require Cloudflare
  build-root configuration for the monorepo subdir and a Node build step in an otherwise Go repo's CI.
  Manual deploy keeps the toolchains cleanly separated and matches existing muscle memory. Can be
  revisited if deploy frequency justifies it.

## Consequences

- A Node/Vite toolchain now lives inside a Go repo, scoped entirely to `web/`. The two build systems
  do not collide because the site build is invoked manually from `web/`.
- The demo GIFs in `docs/` are the single source consumed by both the README and `web/`.
- Publishing a site update is a deliberate `npm run deploy`, not a side effect of pushing — acceptable
  for a low-frequency marketing page.
- If the site ever needs preview deploys per-PR or more frequent updates, switching to Git-connected
  deploy is a new decision, not a reversal of "in-repo."
